**⭐**ipairs与pairs？

⭐结论：遍历数组用 ipairs，遍历字典/不确定结构用 pairs。# 不可靠时用 pairs。

  1. 遍历的顺序 (Order)

这是最关键的区别。

  • pairs 是无序的：它会遍历表中的所有键值对（包括数组部分和哈希部分），但遍历的顺序是不确定的。如果你用 pairs 遍历一个数组，输出的顺序可能和你插入的顺序完全不一样。
  • ipairs 是有序的：它专门用于遍历数组部分。它会严格按照数字索引 1, 2, 3... 的顺序递增进行遍历，保证了输出顺序与索引顺序一致。
  1. 遍历的范围与中断机制
  • pairs 遍历所有：无论是数字索引、字符串索引还是其他类型的索引，pairs 都会把它们全找出来。即使数组中间有 nil（空洞），它也会跳过 nil 把后面的键值对找出来。
  • ipairs 遇到 nil 就停止：它从索引 1 开始遍历，只要遇到索引对应的值为 nil，立刻停止遍历。它完全忽略字符串键（如 name = "张三"）或非连续的数字键（如突然来个 [10] = "hello"，而 2~9 是 nil）。
  1. 性能差异

对于纯数字索引的数组，ipairs 的底层逻辑只是简单的整数递增访问（类似 C 语言的 for (int i=1; ...)），而 pairs 需要遍历底层复杂的哈希表结构。因此，在专门处理纯数组时，ipairs 的意图更明确，底层执行效率通常也略高一点点。

⭐__newindex：赋值不存在的键时重定向

当 __newindex 作为函数时，它其实接收 3 个参数，通常简写为 __newindex(t, k, v)。

它是一个触发器（回调函数）。 当你试图给表中 不存在的键（新键） 赋值时，Lua 就会自动调用这个函数。

这三个参数分别代表：

  • t (table)：正在被操作的表本身。
  • k (key)：你想要添加的“新键”。
  • v (value)：你想要赋给这个新键的“值”。

注意核心条件： 只有当这个键在表里本来不存在（值为 nil）时，才会触发 __newindex。如果这个键已经有值了，你再去修改它，是不会触发 __newindex 的！

Lua 完全忽略 __newindex 函数的返回值。 它不需要你 return 任何东西，它的存在是为了让你在这个函数里面“做点动作”（比如打印警告、把数据存到别的地方、或者报错）。

情况 A： __newindex = {} (备用表模式) 当你给原表添加新键值时，Lua 发现 __newindex 是个表，就会自动把这个新的键值对，存到这个备用表里，原表依然是空的。

情况 B：****__newindex = function(t, k, v) (函数拦截模式) 当你给原表添加新键值时，Lua 会拦截这次赋值，转而去执行你写的函数。 最重要的一点： 一旦触发了你的函数，Lua 就认为它把工作交给你了，原表根本不会被赋值！ 除非你在函数里手动把值塞进去。

Lua中的闭包

  1. 什么是 Upvalue？

普通函数只能用两种变量：自己内部定义的（局部变量），或者别人传进来的（参数）。 但如果你在一个函数内部又写了一个函数，里面的小函数用了外面大函数的变量，这个变量对小函数来说，就叫 Upvalue。

神奇的地方来了：按理说，createCounter 这个大函数执行完后，里面的局部变量 count 应该就 死掉（被销毁） 了。但是！你调用 myCounter() 的时候，它居然还能记住并修改 count 的值！它是怎么做到的？

  1. 底层揭秘：Open 与 Closed 状态

Lua 是怎么做到让这个变量“起死回生”的呢？它使用了两种状态的切换：

  • 状态 1：Open（敞开状态） 当你刚进入大函数 createCounter 时，局部变量 count 就正常待在 “栈（Stack）” 里 此时，内部的小函数如果要用 count，它的 Upvalue 指针就直接指着工作台上的那个 count。这时候它叫 Open 状态。
  • 状态 2：Closed（打包带走状态） 当大函数执行完毕准备退出了（即：离开作用域），工作台（栈）要被清空销毁了。如果真销毁了，小函数不就傻眼了吗？ 此时 Lua 引擎会做一件非常聪明的事：它发现小函数还需要用到 count，于是在销毁工作台之前，偷偷把 count 的值从工作台上复制了一份，打包塞进了小函数自己的“随身背包（内部结构）”里。 同时，把小函数的指针方向调转，不再指着工作台了，而是指着自己的背包。这时候，它就变成了 Closed 状态。
  1. 补充说明
  • 第 1 点： 说的就是上面的过程。如果连闭包（小函数）自己都不被使用了，那么那个 Upvalue 也就没用了，它就会“指向自己”，等垃圾回收车（gc = garbage collection）把它清理掉。
  • 第 2 点： 说的是 Lua 底层为了管理这些跨越作用域的变量，把它们串成了一个链表（linked list），运行小函数的时候，顺着链表去找对应被打包起来的变量。

💡 总结成一句话

闭包就像是一个带着“空间戒指”的函数，当它外面依赖的变量快要因为函数结束而死掉时，Lua 底层会自动把那个变量的尸体（值）装进戒指里带走（从 open 变成 closed），这样闭包以后依然可以随时使用和修改它。

  1. 终极公式：闭包 = 函数 + 引用环境
  • 普通函数就像是一个干活的工人，你给他什么材料（传参），他就干什么活。
  • 闭包则是一个“自带干粮”的工人。它不仅包含干活的步骤（代码），还把自己偷偷藏起来的私人物品（外部环境的局部变量/upvalue）装在背包里带在身上了。
  1. 词法作用域（基础理解）

就是我们上一问提到的例子。bar 是子函数，foo 是父函数。bar 用了 foo 里的变量 i。这就叫“词法作用域”——代码写在哪，就能往外层找变量用。

  1. 面试加分项：为什么说“Lua的所有函数都是闭包”？

直觉误区：很多人以为，只有写了嵌套函数才算闭包，不写嵌套函数就没有闭包。

底层真相：错！ Lua 编译器在运行你的代码前，会自动把整个脚本文件里的所有代码，偷偷包裹在一个叫 main 的隐形大函数里面。

结论：所以，你在脚本最外层写的局部变量，其实都是这个隐形 main 函数的局部变量。你写的任何函数只要用到了它们，本质上就已经发生了跨函数调用。因此在 Lua 中，本质上所有函数全都是闭包！

  1. 闭包在实际开发中有什么用？
  • 作用 1（简洁）： 当我们需要一个一次性的回调函数时，不需要专门去定义一个完整的对象或给函数起个名字，直接写个匿名闭包传进去就行了，代码非常干净。
  • 作用 2（形成独立的运行环境）： 这是闭包最大的用途。它可以用来替代面向对象编程（OOP）。你每次调用父函数生成一个闭包，它都会自带一个全新、独立的变量环境，互相之间不干扰。就像游戏里每个怪物都有独立的血条，可以用闭包来管理这个血条。
